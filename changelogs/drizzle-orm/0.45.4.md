## Breaking change: `binary` and `varbinary` now return `Buffer` by default

**Affects:** MySQL (`drizzle-orm/mysql-core`) — `binary()` and `varbinary()` columns.

### What changed

`binary` and `varbinary` columns now default to `mode: 'buffer'`, returning a `Buffer` from `select` queries instead of a `string`. This matches what the `mysql2` driver actually returns for these column types.

Previously the decoder silently called `.toString()` on the `Buffer` returned by the driver, which corrupted arbitrary binary data. The new default preserves byte-level fidelity.

### Migration

If your existing code reads these columns as strings, add `{ mode: 'string' }` to preserve the old behavior:

```ts
// Before (implicit string mode)
const table = mysqlTable('t', {
  data: binary('data', { length: 16 }),
  payload: varbinary('payload', { length: 255 }),
});

// After — opt back into string mode
const table = mysqlTable('t', {
  data: binary('data', { length: 16, mode: 'string' }),
  payload: varbinary('payload', { length: 255, mode: 'string' }),
});
```

To use the new default Buffer mode explicitly:

```ts
const table = mysqlTable('t', {
  data: binary('data', { length: 16 }),           // Buffer (default)
  payload: varbinary('payload', { length: 255 }), // Buffer (default)
});

// $inferSelect types:
// data:    Buffer | null
// payload: Buffer | null
```

### New `mode` option

| Mode | `$inferSelect` type | When to use |
|------|---------------------|-------------|
| `'buffer'` (default) | `Buffer \| null` | Storing arbitrary binary data |
| `'string'` | `string \| null` | Storing bit-strings or text in binary columns (legacy) |

### Notes

- The encoder always accepts both `Buffer` and `string` regardless of mode — `mysql2` handles both correctly.
- `binary(N)` columns are fixed-length and right-padded with `0x00` bytes by MySQL. When comparing round-tripped values use `.subarray(0, writtenLength)` or compare against a zero-padded buffer.
